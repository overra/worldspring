Worldspring · Development Field Log · Jun 9 – Jul 13, 2026

One month,
one island,
protocol fourteen.

Worldspring is an open-source, browser-native DayZ-like — a low-poly procedural island with zombies, hunger, full-loot PvP, and base building, running as a React Three Fiber client against an authoritative Cloudflare Durable Object with a shared deterministic sim. Thirty-five days ago it was an empty repo. Today it is live at worldspring.games, with a server directory, a release pipeline, physics, vehicles, and a save path measured in single milliseconds at world cap — and, as of this week, a second game running on the same engine, which is rather the point of everything below.

Days 32–35Jul 10–13proto v12–14

Forests and the performance reckoning

Trees grew into a full server-authoritative lifecycle — and two deep performance investigations ended with the server faster than it has ever been, with the telemetry to prove it permanently. A four-PR sweep emptied the audited backlog — client frame wins, an 82% GPU-texture cut, every open review follow-up paid down — and then the render audit's one real project landed: the whole island learned to frustum-cull, walls and the join hitch and input pacing included, closing the audit out. Then the net stack got the same treatment — the per-tick snapshot broadcast, the server's new top cost, was re-encoded to a binary wire 3.5× lighter than its JSON, and with the room's headroom proven the player cap rose to 40.

  • The tree lifecycle
    EZ-Tree forest assets and fracture debris, then server-authoritative planting, wall-clock growth, seeds, and chopping — capped by stumps, persistent axe-breakable trunks, and an 8-variant catalog.
    PR 85PR 86PR 87
  • Asset compression in the pipeline
    gltf-transform pinned into models:export — meshopt + WebP with the structural passes disabled to preserve the node-name contract. Shipped GLBs roughly halved.
    PR 88
  • The save-tick fix: 158 ms → ~1 ms
    The world snapshot split by write cadence — a per-tick row, a trees row, and 64 dirty-gated spatial structure buckets — so an idle world writes ~27 KB instead of re-serializing 564 KB. Verified on prod and at the full 3,000-piece cap.
    PR 89PR 90
  • The tick-cost scare, solved in an afternoon
    The full-cap loadtest showed tick cost 4× the baseline. New per-phase telemetry attributed it in one sample: Rapier was stepping a fully-asleep world four substeps every tick against ~4,000 static colliders. A skip-when-asleep fix (with a wake guard the replay harness caught the need for) brought tick EMA from 12.9 back to 3.7 ms — and the telemetry is now permanent.
    PR 91
  • Client frame wins & the texture diet
    The render audit's Tier-1 batch: N8AO's per-frame transparency detection killed (it was silently buying two extra scene renders every frame since day one), half-res AO on medium, shadow maps frozen at night, honest GPU-tier routing with a one-time re-probe of existing profiles, and idle player rigs detached from the scene graph. Then GPU texture memory fell 95 → 17 MB — fourteen Meshy props were shipping solid-black emissive textures, and the characters carried 1024px palettes that read identically at 256.
    PR 94PR 95
  • Paying down the review ledger
    Every open follow-up closed: vehicles collide with all stumps (natural fells keep a physics stub, pinned by new replay probes), lying trunks are axeable from their ends, the planted-fell stump swap lands in the same frame as its debris burst, Safari no longer hitches building fracture templates — and the loadtest's steady-tick gate now excludes save-contaminated windows instead of trusting the save schedule.
    PR 92PR 93
  • The island learns to cull
    Every world-spanning dressing renderer — trees, planted trees, stumps, structures, scatter, trim, containers — used to be vertex-processed in full every frame, both passes, no matter where the camera looked (~190K forest triangles alone). A shared chunking helper now splits them into per-cell instanced meshes with real bounding spheres, so three culls each chunk per camera — shadows included — with a fog-radius gate on top and a 19-check CI harness pinning the invariants. The idle entity pools (60 zombies, 48 corpses, the deer) also stopped paying per-bone matrix work while empty. This was the hard render gate before any huge world tier ships — cleared.
    PR 96PR 97
  • Finishing the sweep — walls, the join hitch, input pacing
    The render audit closed out. The last world-spanning renderer — merged building walls, roofs, floors and the military perimeter — split into per-cell instanced meshes that cull like the rest of the dressing. The ~42 ms join hitch traced to 132 skinned rigs eager-cloned at mount; the pools now clone lazily on demand, in the same frame, so a join pays only for what is actually near spawn. And input commands stopped riding the display refresh rate: a 240 Hz monitor now sends ~60 a second instead of 240 — a smaller client reconcile replay and less server per-command work, with no protocol change. Two medium items were dispositioned honestly rather than built: instancing the 64 moving physics bodies would have cost more culling than it saved, and joining the survivor's skinned meshes was parked as low-reward bind-pose surgery.
    PR 98PR 99PR 100
  • The broadcast diet & a bigger room
    With the render audit closed, the server's top cost was the per-tick snapshot — built and JSON-serialized per connected client, the only O(players²) work in the tick. Two moves. The snapshot wire was re-encoded as a compact binary frame: entity positions became i16 offsets from the viewer (the ≤220 m interest range buys ~0.8 cm resolution, finer than the old round2), the closed enums packed to bytes — while the reconciliation-critical bits (your own state, events, the fog/tree deltas) stayed byte-exact in a JSON tail, so quantization touches only interpolated transforms. A matched loadtest measured 58.7 → 16.6 KB/s per client — a 3.5× cut — and the codec round-trips in a new CI harness. With the wire lighter and the simulation sitting on ~130× tick headroom, the player cap rose 24 → 40, every room-scaled cap moved in lockstep. Per-entity delta and names-once-on-join — the stateful half — were left for a later pass.
    PR 101PR 102
  • Measuring the ceiling — and declining a lever
    The next queued move was a per-tick spatial grid, to stop the snapshot scan brute-forcing every entity for every client. Before building it, a 40-bot loadtest measured the thing itself. At a full room — 40 players, 60 zombies, ~130 loot — the whole tick ran ~4.6 ms against the 66.7 ms budget, ~14× headroom at the cap, with broadcast ~3.9 ms of it (~80% of the tick). The grid was then declined on the merits: it only skips entities that are out of range, but the case that limits scaling is a clustered one where everything is in range — so the grid would skip nothing and only add bucketing overhead — and every distance check in the whole tick, combined, is under a millisecond even at max density. The real broadcast cost is constructing and serializing the in-range entities for each client, which no grid touches. Delta compression is the one lever that would, so it was reclassified from a server need to mobile-bandwidth work and parked. The server-scaling campaign closes here — the ceiling mapped instead of guessed. No PR: a measurement and a decision.

The turn, built

The platform bet stopped being a document. In a single sweep the engine   game seam was carved, survival was re-expressed as one mode on it, and a second, genuinely non-survival mode — a round-based arena — was built on the same hooks to prove the seam was real rather than decorative. Then the trust layer landed: one command that tells a forked, agent-modded server whether it just broke determinism, the wire, or the sim. Alongside it the storefront and docs shipped their redesign and stopped being deployed by hand. Four of the five foundations named in the turn are now on main.

  • The storefront goes live — and deploys itself
    The site and docs took on the field-manual design language, and the last manual step disappeared: merging to main now ships the game Worker and worldspring.games. The redesign had been sitting merged-but-dark for days because the web deploy was still a hand-run command — the first automated run put it in front of people.
    PR 103PR 104PR 106PR 109
  • The engine ⟷ game seam
    Carved as an internal seam rather than the workspace-package split first imagined — the package boundary would have bought migrations and API lock-in we don’t need while forks sync with the engine not at all. Instead a GameMode object took ownership of what makes this survival: the per-tick gameplay composition, world seeding, and the player lifecycle (where you spawn, what you carry, how long death lasts). The survival tunables moved out to their own module. Every increment was a provable pure move — the flagship’s behaviour is byte-identical, verified by loadtest at each step.
    PR 105PR 110PR 111PR 112
  • Arena — the first game that isn’t survival
    A seam you cannot build a second game on is decoration, so one got built. Arena is a round-based frag deathmatch on the same procedural island: spread spawns, a pistol, first to fifteen kills takes the round, a short intermission, then everyone respawns and the score resets. What makes it a different game lives entirely in its hooks — none of survival’s world ticks at all (no zombies, no hunger or cold, no trees, wildlife, weather, loot or building), and it seeds nothing. A new kill hook, which survival simply ignores, does the scoring. Selecting it is one config field. Its headless probe doubles as the executable template an agent pattern-matches against. The round HUD is still to come — arena is playable today, but the scoreboard rides the UI arc.
    PR 114
  • The trust layer: mod:check
    One command a fork runs before deploying, aimed squarely at the failures that never announce themselves: types, worldgen determinism, the wire round-trip and the sim, then the build. The determinism check is the interesting one. The committed fingerprint is Linux-canonical — V8’s transcendental math drifts by an ULP across machines — so comparing against it on a Mac cries wolf. Instead the world is generated twice and the two runs are required to match: platform-independent, and it catches the actual failure mode, a stray Math.random() or Date.now() leaking into worldgen. The byte-exact cross-machine gate stays in CI where it belongs. Proven by injecting the leak and watching the guardrail trip.
    PR 115
  • A server is a deploy, not a route
    A question about preview deployments — “the subdomain says play, but we only had the one game” — surfaced an assumption worth naming. The worker routes everything to a single Durable Object with one config, so the unit is fixed: one deploy is one server is one game. That settles the rest. Mode is a per-deploy config field, not a route; a first-party arena is simply a second deploy of the same binary; the directory is what scales servers, not a subdomain each; and previews don’t multiply per mode, because correctness is gated by headless probes rather than by standing something up. No PR: a decision, written into doc 00.

The turn ahead

Worldspring began as a DayZ-inspired survival game. A month in, the thing under the hood turned out to be more general than the game on top of it — an authoritative-DO, deterministic-sim, procedural-world, physics-and-building engine, with survival layered on as mostly-optional systems. The presets already hint at it: homestead and driftwood are barely survival at all.

So the aim widens. Worldspring is becoming a platform for browser-native, self-hostable multiplayer worlds — you fork it, point a coding agent at it (“make this a heist, a race, a build-battle”), and deploy to your own Cloudflare. Survival is the flagship and the first mode, not the ceiling. The shape of it is Minecraft or Garry’s Mod server-modding, for the agent era: “write a plugin” becomes “tell an agent what you want.”

The bet is cheap because the hard 90% — the netcode, the world, the physics — is done and genre-agnostic; the work is architecture, docs, and tests that make the flagship better too. And a self-hosted server is fully self-contained: its client and its server ship from one fork, so it needs to sync with the canonical engine not at all — only the directory’s small listing contract. The one thing worth guarding above everything is determinism, the property that makes the netcode work and the thing an agent is most likely to break invisibly; a one-command guardrail suite that catches a desync before deploy is the trust layer of the whole idea.

The near-term foundation was: carve a clean engine   game seam, give it a GameMode boundary, prove it with one non-survival mode, harden the directory contract, and ship the modder’s guardrail command. Four of those five are now built — see The turn, built above. What remains is the directory contract, the one canonical surface a community server must actually honour, and the fork-and-deploy loop that carries someone from “I want a heist game” to a running server on their own account. The full reasoning lives in docs/plans/00.

The road ahead

Grouped by horizon. “Next” items are on the roadmap with their design docs written; “Later” items are deliberately parked with the conditions that un-park them. The server-scaling queue emptied when the ceiling was measured — ~14× tick headroom at the 40 cap — so that work now waits on real demand rather than sitting ready. The engine seam, game modes and the modder’s guardrail have left this list — they shipped (see The turn, built). What leads now is the other half of the platform: the one contract a community server must honour, and the loop that gets a fork deployed.
